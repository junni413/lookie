    ```
    ./data/prod/
    ```
    
- Let's Encrypt SSL 인증서 마운트
- 로그 파일 호스트 저장 ()
    
    ```
    ./data/logs/
    ```
    

### **3.2 배포 순서 (운영 환경)**

1. **MySQL 컨테이너 시작** → healthcheck 대기
2. **Flyway migrate 실행** → DB 스키마 마이그레이션
3. **Backend 시작** (migrate 성공 후)
4. **Nginx 시작** (Backend 준비 후)
5. **Airflow 초기화** (airflow-init)
6. **Airflow Webserver & Scheduler 시작**
7. **(선택) AI Server 시작** ( 사용 시)
    
    ```
    --profile ai
    ```
    

### **3.3 주요 특이사항**

### **Backend**

- **Flyway 비활성화**: 운영 환경에서는 별도 migrate 컨테이너 사용
    
    ```
    
    SPRING_FLYWAY_ENABLED=false
    ```
    
- **프로파일 분리**:  프로파일로 실행
    
    ```
    docker
    ```
    
- **로그 파일**: 로 영속화
    
    ```
    /var/log/app/backend.log
    ```
    

### **AI Server**

- **모델 파일 빌드 시 포함**: Dockerfile에서 COPY로 명시적 포함
- **CPU 전용 PyTorch**: GPU 불필요, 비용 절감
- **SSRF 방어**: 허용된 도메인만 이미지 다운로드 가능

### **Nginx**

- **멀티 스테이지 빌드**:
    1. Node.js로 React 빌드
    2. Nginx에 정적 파일 복사
- **WebSocket 지원**: Upgrade 헤더 처리
- **긴 타임아웃**: WebSocket 연결 유지 (3600초)

### **Airflow**

1. **로그인** → 작업자 계정으로 로그인
2. **출근** → 근무 시작 (work_logs 생성)
3. **작업 시작** → 배치 작업 할당
4. **토트 스캔** → 토트 바코드 스캔 (tote_id 연결)
5. **지번 스캔** → 집품 위치 스캔 (zone_locations)
6. **상품 스캔** → 상품 바코드 스캔 (products)
7. **수량 조정** → 실제 집품 수량 입력
8. **다음 상품** → 다음 batch_task_item으로 이동

### **시나리오 2: 파손 상품**

1. **지번 스캔** → 집품 위치 확인
2. **상품 스캔** → 상품 확인
3. **이슈 신고** → 파손 이슈 생성 (issue_type='DAMAGED')
4. **파손 신고** → 이슈 상세 입력
5. **이미지 업로드** → 파손 증빙 사진 업로드 (issue_images)
6. **AI 분석 대기** → AI 서버로 이미지 전송
7. **결과 조회** → AI 판정 결과 확인 (ai_judgments)
    - PASS: 정상으로 판단 → 작업 계속
    - FAIL: 파손 확정 → 관리자 호출
    - RETAKE: 재촬영 요청
8. **다음 작업 진행** → 판정에 따라 후속 조치

### **시나리오 3: 재고 없음**

1. **지번 스캔** → 집품 위치 확인
2. **이슈 신고** → 재고 없음 선택 (issue_type='OUT_OF_STOCK')
3. **재고 없음 신고** → 이슈 생성
4. **AI 분석** → GMS API로 원인 분석
5. **결과 조회** → AI 판정 결과
    - 원복 대기: 다른 작업자가 반품 중
    - 파손으로 인한 재고 없음: 파손 처리 필요
    - 재고 있음: 위치 오류 가능성
6. **다른 작업 진행** → 해당 아이템 스킵

### **시나리오 4: 관리자 파손 처리**

1. **관리자 화면 조회** → 이슈 목록 확인
2. **긴급 이슈 선택** → priority='HIGH' 이슈
3. **파손 확인** → AI 판정 및 이미지 검토
4. **파손 판정** → 관리자 최종 결정
5. **이슈 해결** → status='RESOLVED', reason_code='DAMAGED'
6. **작업 재개** → 작업자에게 다음 작업 지시

### **시나리오 5: 재고 없음 해결 (관리자 연결)**

1. **지번 스캔** → 집품 위치 확인
2. **이슈 신고** → 재고 없음
3. **재고 없음 신고** → 이슈 생성
4. **AI 분석 결과** → 재고 있음 (위치 오류 추정)
5. **(사전 준비) 관리자 화면** → 이슈 확인
6. **작업자 → 관리자 연결** → WebRTC 통화 요청 (webrtc_calls)
7. **연결 성공** → 실시간 영상 통화
8. **관리자 이슈 해결** → 위치 안내 또는 재고 확인
9. **완료 체크** → 이슈 해결 (status='RESOLVED')
